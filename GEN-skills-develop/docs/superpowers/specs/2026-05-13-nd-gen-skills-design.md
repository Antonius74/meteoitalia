# ND Gen Skills V1 Design

Date: 2026-05-13

## Context

`nd-gen-skills` is a greenfield TypeScript npm package for distributing approved enterprise AI workflow skills into repositories. The package is published as `@nexidigital/nd-gen-skills` and is run through `npx`.

The design is based on `nd-gen-skills-requirements.md`. The existing ignored `dist/gen-skills-1.1.10.tgz` shows earlier distribution concepts, but V1 will not preserve the Python implementation or old command model. It may reuse only compatible ideas such as packaged skills and registry metadata.

## Goals

- Install approved AI workflow skills into Codex and Claude repository-local skill folders.
- Keep the architecture independent from one AI tool.
- Use Superpowers as the V1 provider workflow and preserve its upstream skill folder names unchanged.
- Install one active Nexi runtime variant per repo/tool installation.
- Support substantive runtime guidance for frontend React, backend Java, iOS, and Android.
- Track ownership with a lockfile and managed file hashes.
- Never overwrite unmanaged local skills.
- Manage only a marked root `AGENTS.md` block.
- Use generated bundled registry archives for V1 package resolution.

## Non-Goals

- Artifactory or other remote registry backends.
- User-selected package versions.
- Provider import automation.
- Workflow execution runtime.
- Monorepo or nested `AGENTS.md` support.
- Migration from `.codex` or the previous `gen-skills` package.
- Machine validation of generated workflow artifacts.
- Installer-level Jira, Confluence, or test-management integration.

## Architecture

V1 will be implemented as a root TypeScript package:

```text
package.json
tsconfig.json
src/
  cli/
  registry/
  schemas/
  installer/
  adapters/
  lockfile/
  agents-md/
  hashing/
packages/
  provider/
  contract/
  variant/
  utility/
scripts/
  build-registry.ts
dist-registry/
  index.yaml
  packages/
tests/
```

The core is a contract-first modular CLI. Commands parse inputs, resolve registry packages, validate manifests, compute an install plan, run filesystem safety checks, apply the plan, then write the lockfile and managed `AGENTS.md` block.

Commands must not copy files directly. Shared planning and apply modules own file mutation so `install`, `sync`, `add-skill`, `remove-skill`, and `validate` use the same ownership and hash rules.

Tool adapters stay thin:

- Codex maps skills to `.agents/skills/` and the lockfile to `.agents/nd-gen-skills.lock.yaml`.
- Claude maps skills to `.claude/skills/` and the lockfile to `.claude/nd-gen-skills.lock.yaml`.

Registry resolution, manifest validation, hash calculation, install planning, and `AGENTS.md` rendering are shared across tools.

## Package And Registry Model

`packages/` contains canonical source packages:

```text
packages/provider/superpowers/
packages/contract/nexi-workflow-contracts/
packages/variant/frontend-react/
packages/variant/backend-java/
packages/variant/mobile-ios/
packages/variant/mobile-android/
packages/utility/nexi-jira-summary/
```

Each package contains a YAML `manifest.yaml` validated by TypeScript schemas.

The Superpowers provider package contains the fixed workflow subset declared by its manifest. Provider skill folders keep upstream names and contents unchanged. The implementation plan must include an explicit import/review step before packaging so the copied provider source is approved and the manifest matches the installed folder names.

Variant packages each install exactly one visible Nexi runtime skill:

- `nexi-frontend-react-runtime`
- `nexi-backend-java-runtime`
- `nexi-mobile-ios-runtime`
- `nexi-mobile-android-runtime`

All four runtime skills contain substantive platform guidance in V1.

The contract package installs `nexi-workflow-contracts`, which owns shared workflow contracts, test-design structure, manual tester output, traceability expectations, and final delivery guidance.

The utility package installs `nexi-jira-summary`. It is a real, narrow utility skill. It assumes Jira MCP is available; if the required Jira tools are not available, the skill instructs the agent to stop and ask the user to set up Jira MCP before continuing.

`dist-registry/` is generated by `scripts/build-registry.ts` and is included in the npm package:

```text
dist-registry/index.yaml
dist-registry/packages/provider-superpowers-0.1.0.tgz
dist-registry/packages/contract-nexi-workflow-contracts-0.1.0.tgz
dist-registry/packages/variant-frontend-react-0.1.0.tgz
dist-registry/packages/variant-backend-java-0.1.0.tgz
dist-registry/packages/variant-mobile-ios-0.1.0.tgz
dist-registry/packages/variant-mobile-android-0.1.0.tgz
dist-registry/packages/utility-nexi-jira-summary-0.1.0.tgz
```

The installer consumes registry archives, not raw package source. Registry lookup order is:

1. `--registry`
2. `NEXI_AI_SKILLS_REGISTRY`
3. bundled `dist-registry/`

V1 resolves only `latest`. In the bundled registry, `latest` means approved because only approved package artifacts are included.

## Command Behavior

`install --variant <name>` resolves the default provider, required contracts, and selected variant. Bare `install` fails with a clear `--variant` example. If the same variant is already installed, `install` behaves like `sync`. If another variant is installed, the command fails unless `--replace-variant` is passed.

`sync` updates all installed managed packages to the latest versions from the selected registry: provider, variant, contracts, and utilities.

`add-skill <utility>` installs a utility package independently, even if no runtime is installed yet. If a runtime is installed, the command updates the managed `AGENTS.md` block with the utility description.

`remove-skill <utility>` safely removes only managed utility files, updates the lockfile, and removes the utility from the managed `AGENTS.md` block.

`list` reports the local installation from the lockfile.

`list --available` reports packages from the resolved registry.

`validate` checks installed state without running AI workflows. It verifies the lockfile, tool folders, managed files, managed hashes, provider capabilities required by the variant, runtime references, required contracts, utility skill files, and the managed `AGENTS.md` block.

`validate --ci` treats changed managed files as failures.

Unsupported tools fail with a clear error. The default tool is `codex`.

## Safety And Ownership

The lockfile is the source of managed ownership. It records:

- schema version
- tool
- generator package/version
- selected provider and resolved version
- selected variant and resolved version
- runtime skill name
- installed contracts and resolved versions
- installed utility skills and resolved versions
- managed skills and roles
- managed files and SHA-256 hashes

The installer may mutate only:

```text
<tool-root>/skills/<managed-skill>/**
<tool-root>/nd-gen-skills.lock.yaml
root AGENTS.md managed block
```

If a target skill folder exists and is not managed by the lockfile, the command fails without overwriting it.

If a managed file hash matches the lockfile, it may be updated or removed.

If a managed file hash differs:

- local TTY runs warn and ask for confirmation;
- CI or non-interactive runs fail unless `--force` is passed;
- `--force` overwrites changed managed files but still does not overwrite unmanaged folders.

Obsolete managed files are removed during sync after hash checks. Empty managed folders may be removed. Folders containing unmanaged files are preserved and reported.

## AGENTS.md

The installer manages only this marked block in root `AGENTS.md`:

```md
<!-- nd-gen-skills:start -->
...
<!-- nd-gen-skills:end -->
```

All content outside the block is preserved. If `AGENTS.md` does not exist, the installer creates it with the managed block.

The block points agents to the active runtime skill for implementation, debugging, testing, review, and maintenance work. It also lists installed utility skills with short descriptions.

The block does not mention workflow artifact folders. Provider, runtime, and contract skills own those conventions.

## Runtime Skills And Contracts

The runtime skill is the workflow entry point for implementation, debugging, testing, review, and maintenance work. It is not mandatory for simple read-only questions unless the work becomes behavioral, diagnostic, testing, review, or maintenance work.

Each runtime skill instructs the agent to:

1. Load or use `nexi-workflow-contracts` at the beginning.
2. Inspect repository documentation and build files to discover install, build, run, test, and verification commands.
3. Use Superpowers brainstorming for behavior-changing or creative work.
4. Preserve provider-native artifact locations.
5. Produce Nexi contract artifacts according to runtime and contract guidance.
6. Produce test design before automated TDD starts.
7. Use Superpowers TDD for automated behavior coverage where the environment allows it.
8. Apply variant-specific automated, e2e, and manual testing guidance.
9. Use Superpowers verification before claiming completion.
10. Finish with traceability from requirements to tests and verification evidence.

Variant-specific runtime guidance covers:

- Frontend React: package script discovery, React testing tools, browser/e2e verification, Playwright or equivalent when available.
- Backend Java: Gradle/Maven discovery, unit and integration tests, API contract tests, service/controller boundaries.
- iOS: Xcode scheme discovery, XCTest, UI tests, simulator constraints, manual device-test fallbacks.
- Android: Gradle task discovery, JVM tests, instrumented tests, Espresso/UIAutomator, emulator constraints.

`nexi-workflow-contracts` contains shared templates and rules for command discovery, test matrix design, manual tester scenarios, e2e applicability decisions, traceability, skipped verification rationale, and final delivery output.

Runtime skills reference shared contract guidance instead of duplicating all contract content.

## Validation And Testing Strategy

Unit tests cover pure modules:

- manifest schema validation
- registry lookup order
- registry index parsing
- archive manifest extraction
- hash calculation
- lockfile parsing and rendering
- adapter path mapping
- `AGENTS.md` managed block rendering
- install-plan decisions

Integration tests run the built CLI against temporary repositories and cover:

- basic Codex install
- bare install failure
- idempotent install
- variant replacement guard
- `--replace-variant`
- managed file change detection
- `--force`
- unmanaged collision protection
- utility add/remove
- Claude install
- `list`
- `list --available`
- `validate`
- `validate --ci`

Package content tests run before or during registry build. They verify:

- every package manifest is valid
- required files exist
- provider skill folder names match the provider manifest
- all four runtime skills contain required workflow sections
- `nexi-workflow-contracts` exposes shared contract guidance
- `nexi-jira-summary` stops and asks for Jira MCP setup if Jira MCP tools are unavailable

Minimum implementation verification commands are:

```bash
npm test
npm run build
npm run build:registry
npm pack --dry-run
```

## Resolved Decisions

- Use a clean greenfield TypeScript implementation.
- Put the npm package at the repository root.
- Use generated registry archives as the install source in V1.
- Implement substantive runtime guidance for all four initial variants.
- Implement `nexi-jira-summary` as a real utility skill with a Jira MCP availability gate.
- Keep installer validation focused on structure, integrity, ownership, and references.
- Validate authored skill content through package tests.
- Prompt on changed managed files only for local TTY runs; fail in CI/non-interactive mode unless `--force`.

## Implementation Boundary

This design is ready for an implementation plan. The plan should decompose the work into schema/registry foundations, package authoring, install planning, filesystem safety, adapters, command handlers, tests, and packaging verification.
